# audit/PLAN.md

## Purpose

Define the durable, fail-fast audit surface for the orchestrator. Audit is
the single canonical record of "every transport call and every transcript
boundary" required by the spec's Process Audit goal.

The orchestrator does not implement persistence. It defines the interface,
ships an in-memory reference implementation for tests, and provides
helpers (encoders, dedup keys) so external sinks can plug in.

## Public surface (intent only)

- `Sink` interface:
  - `WriteEvent(ctx, EventRecord) error`
  - `WriteTranscript(ctx, TranscriptRecord) error`
  - `Close(ctx) error`
- `EventRecord` is the durable shape of `events.Event` (see `DESIGN.md`
  §8.2); the runtime `Payload` is JSON-serialized into `[]byte` before
  reaching the sink, the rest of the fields are copied verbatim.
- `TranscriptRecord` per `DESIGN.md` §9.
- `EventType` values mirror `DESIGN.md` §8.1, except chunk-level events
  (`request.text_chunk`) are bus-only and never reach the audit sink.
- `Multi(s ...Sink) Sink` — fan-out helper that writes to all underlying
  sinks. Aborts on the first error to keep semantics fail-fast.
- `MemorySink` — reference in-memory implementation for tests, exposing
  `Events()` and `Transcripts()` accessors.
- Sentinels: `ErrSinkClosed`, `ErrSinkBackpressure`.

## Contract

### Required guarantees

- **Completeness:** Every event in §8.1 of `DESIGN.md` (except
  `request.text_chunk`) must be delivered. Implementations that filter
  records violate the contract.
- **Per-task ordering:** Events for the same `task_id` arrive in the
  order they were generated by the engine. Cross-task ordering is
  best-effort; sinks may reorder unrelated tasks for batching.
- **Synchronous semantics:** `Write*` returns only after the sink has
  accepted durability responsibility. A return without error means the
  record will not be lost across an orchestrator crash, modulo the sink's
  own durability guarantees.
- **Idempotence-friendly:** `EventRecord` carries a unique `EventID`;
  `TranscriptRecord` is keyed by `(TaskID, SessionID, RequestID, Kind,
  Seq)`. Sinks may dedup safely.

### Allowed behavior

- **Backpressure:** A sink may return `ErrSinkBackpressure` when its
  internal buffer is full. The engine treats this as a Request failure
  (the only safe option without queuing the task).
- **Retries inside the sink:** The interface does not forbid an
  implementation from retrying once before returning an error. The
  orchestrator does not retry on top.

### Forbidden behavior

- **Silent loss:** Sinks must not swallow records on full queues. Either
  block until accepted or fail.
- **Reordering within a task:** Implementations must not reorder events
  for the same task, otherwise causal traces (push-before-request) become
  unintelligible.

## Record shapes

### `EventRecord`

| Field        | Type        | Notes |
|--------------|-------------|-------|
| `EventID`    | `string`    | Globally unique; generated by `idgen` |
| `TaskID`     | `TaskID`    |       |
| `SessionID`  | `SessionID` | Empty for `task.*` events |
| `RequestID`  | `RequestID` | Empty for `task.*` and `session.*` events |
| `Type`       | `EventType` | Mirrors `events.Event.Type` (see `DESIGN.md` §8.1) |
| `OccurredAt` | `time.Time` |       |
| `Payload`    | `[]byte`    | JSON serialization of the typed payload (see `events/PLAN.md`) |
| `Schema`     | `string`    | Schema version label (`v1`) for forward compat |

### `TranscriptRecord`

Per `DESIGN.md` §9. Three kinds: `input`, `output`, `event_summary`. The
`Schema` field on the record carries the schema version (`v1`); bump it
in lock-step with the `EventRecord` schema.

The `event_summary` JSON has a stable shape:

```json
{
  "usage": { "prompt_tokens": 0, "completion_tokens": 0, "total_tokens": 0 },
  "finish_reason": "stop",
  "tool_calls": 0,
  "tool_results": 0,
  "error": null,
  "duration_ms": 0
}
```

## Reference: `MemorySink`

- Stores both events and transcripts in slices guarded by a mutex.
- Exposes deterministic accessors for tests: `Events(taskID)`,
  `Transcripts(taskID)`, `Reset()`.
- Reasonable default for unit tests; not for production.

## Edge cases & decisions

- Audit sink failure mid-Request: the invoker fails the Request, which
  fails the session and the task. Operators see the failure both in the
  partial audit log and in the Result error.
- Multi-sink with mixed durability: `Multi` aborts on first error. To get
  best-effort fan-out (e.g., a Datadog forwarder alongside a Kafka write),
  callers wrap the secondary sinks in their own non-blocking adapter
  before composing.
- Schema evolution: bump `Schema` per record kind. The orchestrator never
  reads back from the audit sink (it is write-only from this side), so
  consumers own backward-compat policy.

## Tests to write (no implementation in this pass)

1. `MemorySink` happy path: write events + transcripts, accessors return
   them in insertion order per task.
2. `Multi` aborts on first underlying failure, downstream sinks not
   contacted past the failed one.
3. Concurrent writers under `-race` for `MemorySink`.
4. Closed sink rejects writes with `ErrSinkClosed`.

## Out of scope

- Concrete durable sinks (file, Kafka, Pulsar, SQL). Operators ship those
  in their deployment binary and pass to `Dependencies.AuditSink`.
- Read APIs (querying the audit log). The orchestrator does not consume
  its own audit log at runtime.
- Sampling / retention policies. Implementations decide.
